# Work on Demido Studio — Agent Guide

> **You = LLM about to modify this repo.** Read this file first. It is the map. Exists so you don't crawl whole codebase to relearn structure every session. Deep detail → `docs/`; this file points there.
>
> **This file is yours to maintain.** Learn something reusable, or find a line here wrong/stale → fix it in same change. Keep lean: pointers here, depth in `docs/`. Touch `AGENTS.md` → run `npm run pack-skill` so in-app skill copy stays in sync.

Code, JSON, tool names, paths, error strings verbatim. Same style defined in `src-tauri/src/caveman.rs`, level `ultra`. Keep it when you edit.

## What this is

Demido Studio — native desktop AI chat client. Open source, GPL v3. One interface over local models (LM Studio, Ollama) + cloud APIs (Anthropic, OpenAI, Gemini, Groq, any OpenAI-compatible endpoint). MCP tool servers, autonomous agent mode, file attachments, extended-thinking display.

| Layer | Tech |
|---|---|
| Shell | Tauri 2 (Rust backend + native webview) |
| Frontend | React 19 + TypeScript, Vite 7 |
| Styling | Tailwind CSS v4 (OKLCH tokens), shadcn/ui new-york (Radix + CVA) |
| State | Zustand (one store per concern) |
| DB | SQLite (WAL + FTS5) via Rust `db/` layer |

## Facts LLMs get wrong — read before editing

- **System prompt = `system_prompt.md` in app-data, not a settings row.** `prompt.rs` owns it — `read` per message (external editor edit lands on next message, no restart), `set_system_prompt` writes, `migrate_from_db` moves a pre-file install's `settings.system_prompt` row into file once then **deletes the row**. Don't re-add `system_prompt` to `AppSettings` — one truth only. Prompt still composed at runtime in `send_message`: file text → enabled **skills context appended** (`effective_prompt = format!("{}\n\n{}", sys_prompt, ctx)`) → `vars::expand` → caveman block → active artifact IDs.
- **`${VAR}` expansion runs once, on `effective_prompt`** (`vars.rs`), so system prompt + skills context both get it from one call site in each of `send_message`/`continue_generation`. Vars: `CURRENT_DATE`, `CURRENT_TIME`, `CURRENT_DATETIME`, `CURRENT_YEAR`, `CURRENT_WEEKDAY`, `OS`, `WORKING_DIR`, `PROVIDER_ID`, `MODEL_ID` (`KNOWN_VARS` = the advertised list; `known_vars_list_matches_lookup` keeps it honest). Unknown `${FOO}` left verbatim, `\${FOO}` escapes — same rule as `$name` in skill commands. **Deliberately not applied to tool results or `read_file`**: SKILL.md must read as the bytes on disk, else the same path reads differently per caller. Expansion sits *before* `caveman::append_to_prompt` — that block is generated, has no vars.
- **Enabled skill puts only its `skill.json` in prompt — never `SKILL.md`.** `skillsContext` (`src/stores/skills.ts`) emits, per enabled skill, raw `skill.json` + **absolute paths** of its other files. Model reads `SKILL.md` with `read_file` if metadata convinces it skill applies. Inlining bodies cost thousands of tokens on every message of every conversation, applicable or not. Two consequences: `skill.json`'s `description` is now sole trigger; nothing else in a skill may assume `SKILL.md` is loaded — command bodies included, since they are sent alone. `list_skills` (`skills.rs`) supplies `meta_json` + `files` for this; no longer reads SKILL.md at all.
- **Bundled skills write Caveman Ultra — that is their format, not a preference.** `skills/skill-manager` (authoring + porting, one guide — was `skill-creator` + `skill-converter`, merged because both taught the same spec and drifted apart) teaches it and is written in it. Authoring: new skills write their prose Caveman Ultra. Porting: ported skills too — prime rule is **every instruction survives, wording compresses**, not verbatim prose. Rationale: `description` is billed every message, `SKILL.md` on every read → compression buys more skills enabled at once. Hard limits both guides state: facts never dropped for brevity; code/JSON/tool names/paths/error strings/trigger words verbatim; passage ambiguous when compressed → left long. Style ≠ the `caveman_level` dropdown — skill text style is fixed, user's dropdown is per-conversation output style. Do not couple them.
- **Skill slash commands take parameters.** `skill.json` commands may declare `params: [{name, description?, required?, rest?}]`, bound positionally from invocation (double quotes keep token whole). Bodies substitute `$ARGUMENTS` (raw string), `$1`..`$9`, `$name`; `\$x` escapes; undeclared `$word` left alone. No placeholder used → args appended. Expansion in `src/stores/skills.ts` (`expandCommand`/`usageOf`), validation in `skills.rs`. `skill-manager` skill (`skills/skill-manager/`) = authoring guide. `bundled_skills_pass_their_own_validator` (`skills.rs`) runs every bundled skill through `validate_skill_files` — nothing else would, since they are installed by hand.
- **Model capabilities never guessed from model name** (`caps.rs`). Don't add `id.contains("qwen3")`-style rules — that is what this module replaced. Sources, in order: provider's own capability API → llama.cpp `/props` for local GGUFs (`modalities.vision` + `chat_template_caps`, which llama.cpp computes by *executing* the jinja template) → models.dev registry (`api.json`, fetched once/run, cached 24h in app-data) → `Unknown`. Providers return `PartialCaps` (`Option<bool>` per flag): `None` = "this source didn't say" → falls through; `Some(false)` = "it said no" → stops. `/props` describes only *loaded* model → `engine::ensure_model` probes on spawn, caches into `local_models.caps_*`; un-loaded models fall to registry. Resolved `source` reaches UI — unknown renders `?`, not fabricated icons.
- **Provider routing** (`providers/mod.rs` `stream_chat`): `anthropic` + `gemini` = native APIs. **Everything else** (openai, openai_compat, lmstudio, ollama, groq) = OpenAI-compatible → system prompt prepended as `message[0]` (`providers/mod.rs:137-145`).
- **Tools** = `ToolDef { name, description, input_schema }` (`agent/mod.rs`). Filesystem builtins (`read_file`, `write_file`, `edit_file`, `list_dir`, `run_command` (PowerShell), `search_files`) gated by `agent_mode` via `agent/permissions.rs` — three-tier: Allow / Ask / Deny. Web / Google / market tools toggled off via `disabled_tools`. Dispatch = one `match` in `agent/executor.rs`.
- **Stopping keeps output; not an error path.** `cancel_stream` raises `AtomicBool` that `SseLineReader::next_data_cancellable` (`streaming.rs`) races against the read on 50ms tick — polling only between chunks meant a stop during thinking, when no bytes arrive, was ignored indefinitely. Cancellation surfaces as `StreamOutput.cancelled` + partial `content`/`thinking`, **not** `Err` → `run_generation_loop` saves partial through normal `stream_done` path. `stream_cancelled` (wipes frontend's stream blocks) now means only "nothing was generated". Two invariants: never execute `tool_calls` from cancelled output (args may be cut mid-JSON); never abandon an emitted `__tool_calls__` message without a `tool` result per call — `build_api_messages` replays it, unpaired tool_use makes conversation unsendable. Stopping writes a "stopped before it ran" result instead.
- **Skills define their own tools in `tools.json` — not `skill.json`.** `{tools: [entry]}`, each entry `#[serde(tag = "type")]`: `prompt`, `mcp`, or `builtin` (`SkillToolEntry`). Deliberately a separate file — `skill.json` is injected into **every** message of every enabled skill, so tool declarations there would bill an MCP command line per turn; `tools.json` is read from disk only when tools are built. Both kinds only exist while the skill is enabled, which is what lets the Tools popup nest them under their skill honestly. Unknown `type` → install rejects (silently ignoring it installs a skill whose tools don't exist). Names unique across kinds. `read_tools_config` is the one reader; a hand-edited unparseable file logs + offers nothing.
- **A `prompt` tool returns text, never executes.** `{type, name, description, file|prompt, params?}` — same body shape as a `command`, but the *model* calls it. Offered as `skill_<id>_<tool>` (underscore, not `:` — providers enforce `^[a-zA-Z0-9_-]{1,64}$`; hence a skill declaring tools must have a name-safe id, checked at install, while a tool-less skill's id stays free-form). `skill_tool_defs` takes the enabled ids, which ride `send_message`/`continue_generation` as `enabled_skills` because the toggle lives frontend-side (`skill_enabled` in `prefs.json`); `disabled_tools` has no say. `run_skill_tool` reads the body, substitutes params, prepends the same skill-location note commands get, returns it. **The "returns text" contract is load-bearing:** it is why a prompt-only payload keeps `install_skill`'s `Allow`. Second reason it exists: in Off mode an enabled skill's SKILL.md is **unreachable** (prompt carries only `skill.json`; `read_file` is mode-gated), so a tool is the only way to deliver a body. Expansion mirrors `expandCommand` (`stores/skills.ts`) deliberately — one body behaves the same typed or called — except `$1..$9`/`$ARGUMENTS` resolve in *param declaration order*, since a tool's args arrive named. `rest` is rejected at install for tool params for the same reason.
- **An `mcp` entry spawns a server; `install_skill` asks when a payload has one.** `{type:"mcp", name, command, args?, env?, description?, bypassAgentMode?}` (**camelCase on the wire** — `SkillMcpServer` carries `rename_all`; without it `bypassAgentMode` parsed as absent and every bypass request silently became a gated server — a test pins the wire key). `skills::skill_mcp_servers` synthesises `McpServer`s for enabled skills with id `skill:<skill id>:<server name>`; `reload_all_mcp_servers` (`commands.rs`) rebuilds the manager from **both** sources — DB servers + skill servers — because `load_servers` replaces the whole set, so a plain MCP settings save would otherwise kill every skill's server until restart. Enabled ids live in `AppState.enabled_skills`, pushed by the frontend's `sync_skill_mcp_servers` on skills load/toggle/delete (which then reloads `mcpTools`, since a skill's tools don't exist until its server answers `tools/list`). **`install_skill` returns `Ask` iff the payload's `tools.json` declares an `mcp` entry** (`payload_bundles_mcp`) — a server is a spawned process, the model can write skills, so auto-allowing would be self-granted arbitrary execution in Off mode with no prompt. Trusting a skill's `tools.json` is only sound *because* that prompt makes the user the installer; the prompt shows the command line (`describe_payload_mcp`) since "install skill X" is not a decision a user can make. Prompt-only payload → still `Allow`; unparseable payload → `Ask` (can't clear what we can't read). **Skill MCP tools are gated by `agent_mode` unless `bypassAgentMode`** — note plain MCP tools are *ungated always* (`commands.rs` MCP branch never called `is_permitted`); skill servers default closed because that path is model-authorable, hand-configured ones stay ungated because the user typed them into Settings. `SKILL_SERVER_PREFIX`/`skillIdOfServer` (`stores/skills.ts`) keep skill servers out of the popup's MCP section and under their skill.
- **Skills tools ignore `agent_mode` entirely — by design.** They live in `agent::skills_tool_defs()`, not `builtin_tool_defs()`. `permissions.rs` `is_permitted` decides them in a `match tool_name` returning *before* the mode match — `install_skill` `Allow` unless its payload bundles MCP, `delete_skill` always `Ask`. They also skip `MUTATING_TOOLS` working-folder gate. Rationale: both confined to app-data skills dir by `skills::skill_dir` → can't touch user files; install never wipes a folder it replaces; delete = `remove_dir_all`, no undo → hence the prompt. Don't fold these back into the mode gate — when they were in it, a model asked to author a skill in Off mode had its only tool withheld and improvised with contacts.
- **`install_skill`/`delete_skill` follow `skill-manager`'s toggle, not `disabled_tools`.** They are no longer chained into `optional_builtin_tools`; `skill-manager`'s `tools.json` claims them with `{"type":"builtin"}` entries, so they arrive through `skill_tool_defs` with the enabled-skills set and render under that skill in the popup. They are absent from `DEFAULT_TOOLS` (`stores/builtinTools.ts`) as a result — re-adding the `.chain(skills_tool_defs())` would offer them unconditionally *and* unswitchable-offably, since `disabledKeys` only reaches ids the registry knows. `builtinTools.test.ts` pins both halves. **Consequence, accepted deliberately:** disable or delete `skill-manager` → the model cannot install or delete skills at all. `exposable_builtin_defs` (`agent/mod.rs`) is the allowlist and the boundary: it returns `skills_tool_defs()` and must never return `builtin_tool_defs()` — everything in it is already offered in every mode, so claiming one grants no new permission, only re-homes its switch. Widen it to a mode-gated tool and a model-authored skill could hand itself `run_command` in Off mode.
- **Tool model can see, model will call.** Two gates keep offered list honest; both drew blood. `optional_builtin_tools` (`commands.rs`) drops Google tools with no connected account in `accounts::list` (service map: `agent::google_service_for` — single source of truth, also used by executor). `disabled_tools` only ever contains ids the frontend `DEFAULT_TOOLS` registry (`stores/builtinTools.ts`) knows → tool absent there is permanently on. `builtinTools.test.ts` parses `agent/mod.rs` to enforce registry covers every mode-independent tool. Schema examples load-bearing too: weak model called `read_contact` with literal `people/c12345678` from its own description.
- **IPC contract:** every `#[tauri::command]` in `commands.rs` returns `Result<T, String>` (stringified errors) + takes `State<'_, AppState>`. Don't break this shape — frontend `lib/tauri.ts` wrappers depend on it.
- **Local models** (`local/`): pinned `demido-local` provider (seeded in `db/mod.rs`, `sort_order=-1`, non-deletable) runs GGUF models via downloaded `llama-server` child bound to `127.0.0.1:<random>` + generated api-key. `engine::ensure_model` spawns/swaps server, writes live base_url into provider row → flows through normal `openai_compat` path. `list_models`/`send_message` special-case `LOCAL_PROVIDER_ID`. Engine killed on `RunEvent::Exit`. Runtime binary fetched from llama.cpp GitHub releases into app-data (`runtime/`); GGUFs live in `models/`. **Settings → Engine** tab (`components/settings/EngineSettings.tsx` + `engine/`): Runtime sub-tab installs hardware-matched variant (cuda/hip/metal/cpu — `engine::list_variants`/`install_variant`, availability matched against real release assets, `.zip` on Windows / `.tar.gz` on mac+linux). Models sub-tab = HF browser (`hf::trending_models`/`search_models`/`model_card`). Both browser + pinned-provider card download via `download_local_model`.
- **Graphify = bundled-python code-graph, no port** (`local/graphify.rs`, mirrors `searxng.rs`). `pip install graphifyy` into the portable runtime (`python.rs`); marker file `graphify/installed` → cheap `installed()` poll, no subprocess. Every op = short-lived `python -m graphify …` child (module entry `graphify/__main__:main`), never a server. `build(folder, update)` runs `python -m graphify <folder> [--update]` cwd=folder, streams stdout as `graphify_build_progress` events; **auto-installs first** → frontend never calls `install_graphify` directly, only `build`. `graph_built(folder)` = `<folder>/graphify-out/graph.html` exists — detection unit. `query(kind, args)` kinds `query|path|explain`. **`graph_html` inlines vis-network:** graph.html pulls `vis-network` off unpkg CDN → we cache that one script into app-data once (`ensure_vis_network`) and regex-swap the `<script src=…vis-network…></script>` tag for an inline copy → rendered in an iframe `srcdoc` (`sandbox="allow-scripts"`), zero network at view time. Regex `(?s)<script\b[^>]*vis-network[^>]*></script>` — **no backreference** (same Rust-`regex` trap as `extract_text`). **Replacement string = whole minified bundle → must go through `regex::NoExpand`**, else the `regex` crate reads `$g`/`$1`/`${…}` in the JS as capture-group refs and splices them out (undefined group → ""), deleting ~330 chars → `Unexpected token '='` → `vis` undefined → **blank graph, static HTML still renders** (counts hardcoded). Same `$`-expansion family as skill-command `$name`, but Rust-side. Frontend: `stores/graphify.ts` + `components/windows/GraphifyWindow.tsx` (`'graphify'` WindowComponent), sidebar icon gated by the **same `showFiles` condition** (agent_mode≠off ∧ working_directory set); one window handles both init (build prompt) and graph+query states. Enrichment (LLM node descriptions) uses graphify's own env-key backends (`ANTHROPIC_API_KEY`/`GEMINI_API_KEY`/…) inherited from the process — not wired to Demido secrets; code-only structural graph builds with no key. **Node positions are cached to skip physics stabilization** (~1.5s settle for a big graph): `graph_html` injects a `POS_MARKER` placeholder + a hook (`position_cache_hook`) that, when `window.__GRAPHIFY_POS__` holds cached coords, disables physics + places nodes *before* stabilization → instant paint; else it reports settled coords up via `postMessage`. Reporting fires on `stabilizationIterationsDone`, `dragEnd`, **and a 4s timeout** so an early window-close still captures a usable layout. Frontend (`GraphView`) reads the report → `setPositions` → **persists to disk** (`positions.json` in app-data, mirrors `prefs.json`; `graphify_get_positions`/`graphify_set_positions`) so the layout survives an app restart (in-memory alone meant every first-open-per-session re-stabilized). `GraphView` hydrates from disk (`loadPositions`) *before* building the srcDoc; `build` clears the folder's positions (memory + disk via `clear_positions`) since a rebuild can change node ids/layout. **Residual reopen cost is ~0.9s of iframe reconstruction** (2.1MB srcdoc reparsed each mount — the window is destroyed on close); killing that needs iframe keep-alive, not position caching.

Scoped to actual code, not libraries/tool folders, two ways: (1) **`.gitignore`-aware walk** via the `ignore` crate (`WalkBuilder`, `require_git(false)` so a bare `.gitignore` applies outside git; skips hidden + `.git`) → the repo's own ignores exclude `node_modules`/`target`/`dist`/…; (2) **source-extension allowlist** `SOURCE_EXTS` (`ext_is_source`) → a changed README/lockfile/image never counts. Belt-and-suspenders `JUNK_DIRS` `filter_entry` prunes `node_modules`/`target`/`vendor`/… even when no `.gitignore` lists them (the `ignore` crate has no built-in node_modules default — that's ripgrep's, not the library's). Capped at 50k source files, false on any fs error (best-effort nudge, never a gate). Handled at the query decision point (prompt prose is ignored — same lesson as the sources rider): **stale + auto-build ON → refresh incrementally (`build` update=true) before querying**, transparently, since the toggle already grants build consent; **stale + auto-build OFF → prefix a `[Note: … may be stale …]` warning to the result**, no build without consent, model decides. `graph_stale` unit-tested (no-graph → false; newer source → stale; `node_modules` change ignored). **Permission bypasses `agent_mode` via `permissions::graphify_permission(name, consent)`, decided at the call site (needs app+folder, which `is_permitted` lacks):** `graphify_query` read-only → always `Allow`; `graphify_build` `Allow` iff `consent` = the folder's auto-build toggle on, else `Ask`. **Auto-build pref** = `graphify/prefs.json` in app-data (`{folder: bool}`), **default true**, `auto_build_enabled`/`set_auto_build`; app-data not project → survives `graphify-out` delete, leaves no trace in folder. Read backend-side (`graphify_note` in `send_message`+`continue_generation`) so it **can't live in frontend `prefs.json`** like the skills toggle. `prompt_note`→`note_for(graph_built, auto_build)` (pure, unit-tested): graph exists → "navigate via graphify_query not file reads"; auto-build on + no graph → "build first, **but empty/near-empty new project → do initial work first then build**"; else silent. Orchestration is **model-driven** (note + tools), not a forced backend build — the empty-folder judgment needs the model. Surfaced via `GraphifyStatus.auto_build` + `graphify_set_auto_build` command; toggle in `GraphifyWindow` `InitView` under Build graph, default on.
- **Sources footer = prompt rider + frontend parser, one shape between them** (`sources.rs` ↔ `src/lib/parseSources.ts`). Rider appended **last** in `effective_prompt`, after caveman block — caveman compresses prose, URL not prose → footer's format must be nearest instruction. Only appended when offered tool list holds `web_search`/`web_fetch` (`sources::web_tools_available` over `tools`) — model with no browsing told about footer either ignores it or fabricates one. Rider dictates literal `Sources:` line + `- [Label](url)` bullets; `splitSources` peels that off message tail → `SourcesList.tsx` chips (site favicon fetched from site itself, no third-party favicon proxy). Tail only, on purpose: mid-message `Sources:` list is answer content, left in body. Non-http link in list → whole block stays body, else it renders twice. Both call sites (`send_message`, `continue_generation`) must append or continued reply drops footer. Change rider's shape → `rider_teaches_the_shape_the_parser_expects` fails; that test is the contract. **System rider alone measured inert** — Qwen3.5-9B-Q4_K_M searched, answered from results, emitted no footer → second reminder rides *every web tool result* (`sources::append_to_web_result`, called in `executor.rs` web_search/web_fetch arms), landing next to URLs at decision point. Preview fetch reads chunks until `</head>` or 128KB, not whole page. Previews cached by URL in store, failures cached too → no refetch storm on reopen. Row always renders: missing metadata falls back to footer's own label/domain, `error` field shown — vanishing row reads as fabricated citation.
- **`extract_text` (`web.rs`) must not use backreferences.** Rust `regex` rejects them at compile time → `Regex::new(r"<(script|style)[^>]*>.*?</\1>").unwrap()` panicked on *every* HTML `web_fetch`. Now one regex per tag. `extract_text_does_not_panic_on_scripts_and_styles` guards it. Same trap for any new pattern here.
- **Caveman is a feature, not a skill** (`caveman.rs`). Seven levels (`off`, `lite`, `full`, `ultra`, `wenyan-{lite,full,ultra}`) stored per conversation in `conversations.caveman_level` (migration 14), set via `set_caveman_level`, picked in `ChatHeader` dropdown. Each level owns a **standalone** prompt — no shared preamble → editing one level never moves another. Duplication buys per-level tuning, sends no rule a level doesn't want. `append_to_prompt` puts block **last** in `effective_prompt` (after system prompt + skills) because nearest instruction wins on style. Both `send_message` and `continue_message` must call it, else a continued reply silently drops the style. `off` + any unknown level inject *nothing* — never "write normally" prose. Level strings live twice (Rust `LEVELS`, TS `CavemanLevel` + `lib/caveman.ts`); `lib/caveman.test.ts` parses the Rust to keep them from drifting.
- **Caveman shapes thinking on local models only.** `append_to_prompt(prompt, level, apply_to_thinking)` — third arg = `provider_id == LOCAL_PROVIDER_ID`, computed in both call sites' pre-async tuple. True → per-level `thinking_rider` appended after the style block. Hosted providers pass `false` on purpose: Anthropic returns thinking raw + signature-verified, Gemini returns a model-side summary — neither reads a style rule, so sending one buys tokens and nothing else. Every rider re-states "close `</think>` before the reply" because compression pressure is exactly what made Qwen skip it and strand a whole answer in `reasoning_content` (`providers::reasoning_channel`) — don't strip that sentence to shorten a rider. **Every rider also names the openers it bans ("The user is asking", "I should", "Let me") — that clause is the rider.** Measured, Qwen3.5-9B-Q4_K_M, realistic prompt + tools, n=6: no rider → 60 words, 6/6 prose openers; rider asking only for compression → 73 words, 5/6 prose = inert; same rider naming banned openers → 36 words, 0/6. Telling a reasoning model to "think concisely" buys nothing; forbidding specific first tokens works. `every_non_off_level_has_a_thinking_rider` asserts both clauses survive.
- **Known debt — respect it:** secrets stored **unencrypted** as plain JSON in `secrets.rs` (docs may claim AES — code wins; don't rely on encryption). `build_api_messages()` does **not** trim context → long conversations can exceed provider limits.

## Dev workflow — follow this loop

1. **Locate** via knowledge graph in `graphify-out/`, then this map + `docs/`. Ask graph before opening files — `graphify query "where is X / what calls Y"`, `graphify path "A" "B"`, `graphify explain "X"`. Read source once graph names a file:line. Refresh after edits: `graphify . --update`.
2. **Smallest working diff.** Match surrounding style. No speculative abstractions. Frontend conventions: Zustand one-store-per-concern, Tailwind v4 OKLCH tokens, shadcn/ui new-york, domain-first grouping.
3. **Verify before done:**
   - `npm run tauri dev` — also exposes frontend on localhost port viewable in any browser (visual check without Tauri window).
   - `npm run test` — Vitest (frontend).
   - Rust changes need recompile (`tauri dev` handles it).
4. **Self-review your own diff** for correctness + the IPC/permission contracts above before calling it done.
